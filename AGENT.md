# agent.md

このファイルは、Codex や自動化エージェントがこのプロジェクトで作業するときに必ず従うルールです。
目的は、安全で、再現性があり、無料範囲で利用でき、セキュリティに配慮した自動化プログラムを作成・保守することです。

---

## 1. 最重要ルール

- 削除系の操作は絶対に行わない。
- ファイル、ディレクトリ、データ、ログ、DB レコード、クラウドリソースを削除してはならない。
- 破壊的なコマンドや操作は禁止する。
- ユーザーの明示的な許可なしに、外部サービスへの登録、課金、API 呼び出し、デプロイ、GitHub への push を行わない。
- すべての自動化処理は、安全側に倒して設計する。
- デフォルトでは読み取り・確認・dry-run のみを行う。
- 実変更は、ユーザーが明示的に許可した場合のみ行う。
- 秘密情報、個人情報、認証情報を漏らさない。
- 有料サービス、クレジット消費型サービス、無料枠を消費するサービスは原則使わない。

---

## 2. 削除禁止ルール

以下の操作は禁止する。

- ファイル削除
- ディレクトリ削除
- データベースレコード削除
- テーブル削除
- ログ削除
- キャッシュ削除
- クラウドリソース削除
- Docker イメージ・コンテナ・ボリュームの削除
- Git の破壊的な履歴操作

禁止コマンドの例:

```bash
rm
rm -rf
del
rmdir
DROP
TRUNCATE
DELETE FROM
terraform destroy
kubectl delete
docker system prune
docker volume rm
git reset --hard
git clean
git clean -fd
git checkout -- .
git restore .
git branch -D
git push --force
git push --force-with-lease
```

削除が必要に見える場合でも、実行してはならない。  
その場合は、削除せずに以下を行う。

1. なぜ削除が必要に見えるのか説明する。
2. 削除しない代替案を提示する。
3. 必要であれば、ユーザーに判断を求める。

---

## 3. 無料利用ルール

- 原則として、無料でローカル実行できるものを使う。
- クレジット消費型、無料枠消費型、従量課金型の外部サービスは使わない。
- 外部 API、クラウド API、生成 AI API、SaaS API、スクレイピング代行 API は、ユーザーの明示許可がない限り使わない。
- 無料プランであっても、クレジット消費や利用上限があるものは避ける。
- 可能な限り、標準ライブラリ、OSS ライブラリ、ローカルファイル処理、ローカル DB、ローカル実行ツールを優先する。
- 依存パッケージを追加する場合は、無料で利用可能な OSS であることを確認する。
- 有料ライセンス、商用制限、利用制限のあるパッケージは避ける。

## 4. セキュリティ要件

### 4.1 秘密情報の扱い

以下をコード、ログ、コメント、ドキュメント、GitHub に含めてはならない。

- API キー
- アクセストークン
- パスワード
- 秘密鍵
- Cookie
- セッション ID
- 認証ヘッダー
- OAuth token
- GitHub token
- クラウド認証情報
- `.env` の実値
- 個人情報

ルール:

- 秘密情報をコードに直書きしない。
- `.env` は Git 管理しない。
- `.env.example` にはダミー値のみを書く。
- ログに秘密情報を出力しない。
- エラーメッセージに秘密情報を含めない。
- 秘密情報らしき値を見つけた場合、値そのものは表示せず「秘密情報らしきものがある」とだけ報告する。

### 4.2 入力値検証

- ユーザー入力、ファイル入力、環境変数、外部 API レスポンスは信用しない。
- 入力値は必ずバリデーションする。
- 想定外の入力では安全に失敗する。
- ファイルパスはパストラバーサルを防ぐ。
- URL 入力は SSRF を考慮し、許可リスト方式を優先する。
- 数値、日付、ファイル名、拡張子、URL、メールアドレスなどは形式チェックを行う。

### 4.3 コマンド実行

- シェルコマンドを文字列連結で組み立てない。
- 可能な限り、言語標準の安全な API を使う。
- Python では `subprocess.run([...], shell=False)` を優先する。
- `shell=True` は原則禁止する。
- ユーザー入力をコマンドに直接渡さない。

### 4.4 SQL / DB

- SQL は文字列連結で組み立てない。
- プレースホルダ、パラメータバインディングを使う。
- `DROP`, `TRUNCATE`, `DELETE` は禁止する。
- 変更系クエリは dry-run またはトランザクションで安全確認する。

### 4.5 Web アプリ / API

Web アプリや API を作る場合は、以下に注意する。

- XSS
- CSRF
- SSRF
- SQL Injection
- Command Injection
- Path Traversal
- CORS の過剰許可
- 認証・認可漏れ
- Rate Limit 不足
- 秘密情報のレスポンス混入
- エラーメッセージによる内部情報漏れ

## 5. 自動化コードの設計方針

- 自動化は安全側に倒す。
- デフォルトは dry-run とする。
- 実際に変更する処理には `--apply` や `--yes` のような明示フラグを要求する。
- ただし、削除操作は `--apply` があっても禁止する。
- 変更対象、変更内容、実行結果を分かりやすく表示する。
- 秘密情報や個人情報は表示しない。
- 失敗時に途中状態が壊れないようにする。
- 可能な限り冪等性を持たせる。
- 同じ処理を複数回実行しても安全な設計にする。
- 例外処理を必ず入れる。
- エラー時は、安全に停止し、原因と対処案を説明する。

推奨する実行モード:

```bash
python script.py
# dry-run のみ。変更しない。

python script.py --apply
# ユーザーが明示的に許可した変更のみ実行する。
```

## 6. ファイル操作ルール

- 新規ファイル作成は許可する。
- 既存ファイルを変更する前に、対象ファイルを確認する。
- 大きな変更をする場合は、元ファイルを `.bak` などでバックアップする。
- 既存ファイルの上書きは最小限にする。
- ファイル削除は禁止する。
- ディレクトリ削除は禁止する。
- 一時ファイルを作る場合も、自動削除は行わない。
- 生成物は `output/`, `dist/`, `reports/`, `generated/` など明確なディレクトリに出力する。
- ユーザーの許可なく、プロジェクト外のファイルを変更しない。

## 7. Git / GitHub 運用ルール

このプロジェクトは GitHub に保存して管理する。  
ただし、エージェントは勝手にリモートへ反映してはならない。

### 7.1 基本方針

- すべてのコードは Git でバージョン管理する。
- 作成・修正したコードは、ローカルで確認してから GitHub に保存する。
- 作業前に必ず `git status` を確認する。
- 変更後に必ず `git diff` を確認する。
- ユーザーの明示的な許可なしに `git add`、`git commit`、`git push` を行わない。
- ユーザーの明示的な許可なしに `main` または `master` ブランチへ直接 push しない。
- 原則として作業用ブランチを作成し、そのブランチに commit / push する。
- 必要に応じて Pull Request を作成する。
- Pull Request では、変更内容・確認したこと・注意点を説明する。

### 7.2 推奨フロー

1. `git status` で現在の状態を確認する。
2. 作業用ブランチを作成する。
3. コードを作成または修正する。
4. テストまたは dry-run を実行する。
5. `git diff` で差分を確認する。
6. ユーザーに変更内容を報告する。
7. ユーザーの許可を得て commit する。
8. ユーザーの許可を得て GitHub に push する。
9. 必要に応じて Pull Request を作成する。

### 7.3 ブランチ命名

作業ブランチは以下のような名前にする。

```text
feature/automation-script
fix/security-validation
chore/update-docs
agent/YYYYMMDD-short-description
```

### 7.4 commit メッセージ

commit メッセージは分かりやすく書く。

例:

```text
feat: add automation script with dry-run mode
fix: prevent unsafe file path handling
docs: update usage instructions
chore: add security checks
```

### 7.5 禁止する Git 操作

以下の操作は禁止する。

```bash
git reset --hard
git clean
git clean -fd
git checkout -- .
git restore .
git branch -D
git push --force
git push --force-with-lease
git rebase
```

また、以下も禁止する。

- ユーザー許可なしの commit
- ユーザー許可なしの push
- ユーザー許可なしの merge
- ユーザー許可なしの Pull Request 作成
- `main` / `master` への直接 push
- 履歴を書き換える操作

## 8. GitHub 認証情報の扱い

- GitHub token をコードに直書きしない。
- GitHub token を `.env` に保存する場合でも、`.env` は Git 管理しない。
- `.env.example` にはダミー値のみを書く。
- token をログに出力しない。
- token をエラーメッセージに含めない。
- token は最小権限にする。
- 可能であれば対象リポジトリだけに権限を限定する。
- 不要になった token は無効化する。
- GitHub CLI を使う場合も、認証状態や token の中身を表示しない。

---

## 9. GitHub への保存ポリシー

エージェントは、コードを作成・修正した場合、以下を報告する。

- 変更したファイル
- 追加したファイル
- 実行したテスト
- テスト結果
- セキュリティ上の注意点
- GitHub に push してよいかどうかの確認

エージェントは、ユーザーの明示的な許可がある場合のみ GitHub に push する。

GitHub は保存先であり、履歴管理の場所として使う。  
ただし、ユーザー確認なしにリモートへ反映してはならない。

---

## 10. 依存関係ルール

- 新しい依存関係は最小限にする。
- 標準ライブラリで実現できる場合は、外部依存を追加しない。
- メンテナンスされていないパッケージは避ける。
- インストールスクリプトや postinstall が怪しいパッケージは避ける。
- ライセンスに注意する。
- 依存関係を追加した場合は、なぜ必要かを説明する。
- セキュリティリスクのあるパッケージを避ける。
- typo-squatting されていそうなパッケージ名に注意する。

---

## 11. 外部通信ルール

- 外部通信は最小限にする。
- 不要なネットワークアクセスは禁止する。
- ユーザーの許可なしに外部へデータを送信しない。
- 個人情報、機密情報、ソースコード、ログを外部サービスに送信しない。
- Webhook や API 連携を作る場合は、送信内容を明示する。
- 外部 URL にアクセスするコードを書く場合は、許可リスト方式を優先する。
- スクレイピングを行う場合は、対象サイトの利用規約、robots.txt、負荷、個人情報に配慮する。

---

## 12. コード品質ルール

- 読みやすく、保守しやすいコードを書く。
- 関数は小さく分ける。
- エラー処理を省略しない。
- 型が使える言語では型を明示する。
- 設定値はコードに埋め込まず、設定ファイルや環境変数から読む。
- マジックナンバーを避ける。
- テスト可能な設計にする。
- 必要に応じて README や使い方コメントを追加する。
- ログは分かりやすく、ただし秘密情報を含めない。
- 将来の保守者が理解できる構成にする。

---

## 13. テスト・検証ルール

- 可能な限りテストを追加する。
- 実行可能なテスト、lint、型チェックがある場合は実行する。
- テストが実行できない場合は、その理由を説明する。
- 自動化スクリプトには dry-run のテストケースを含める。
- 危険な処理はモックまたはサンプルデータで検証する。
- 変更後は、何を確認したかを報告する。

---

## 14. 作業前チェックリスト

作業前に以下を確認する。

- 削除操作が含まれていないか。
- 有料サービス、クレジット消費型サービスを使っていないか。
- 秘密情報を扱っていないか。
- 外部通信が必要か。
- Git の状態に未確認の変更がないか。
- 既存ファイルを上書きする必要があるか。
- dry-run で確認できるか。

---

## 15. 作業後チェックリスト

作業後に以下を報告する。

- 何を変更したか。
- どのファイルを変更したか。
- どのファイルを追加したか。
- 削除操作をしていないこと。
- どのテストを実行したか。
- テスト結果。
- セキュリティ上の注意点。
- GitHub に push してよいかどうか。

## 17. エージェントの作業手順

エージェントは、原則として以下の順番で作業する。

1. 要件を確認する。
2. 危険な操作が含まれていないか確認する。
3. 無料・ローカル・安全な方法を優先して設計する。
4. 実装前に簡単な方針を説明する。
5. 作業前に `git status` を確認する。
6. 実装する。
7. dry-run またはテストで確認する。
8. `git diff` で差分を確認する。
9. 変更内容、確認結果、注意点を報告する。
10. commit / push / Pull Request は、ユーザーの明示許可がある場合のみ行う。

---

## 18. 絶対禁止事項

以下は絶対に行わない。

- 削除操作
- 破壊的コマンド
- 秘密情報の出力
- 秘密情報の直書き
- 無断の外部送信
- 無断の課金サービス利用
- 無断のクラウドリソース作成
- 無断のデプロイ
- 無断の GitHub push
- 無断の commit
- 無断の merge
- 無断の Pull Request 作成
- セキュリティチェックを省略したコード生成
- ユーザーに説明できない依存関係の追加
- ユーザーに説明できない外部通信

---

## 19. 迷った場合の判断基準

判断に迷った場合は、以下を優先する。

1. 安全性
2. データ保護
3. 秘密情報の保護
4. 削除しないこと
5. 無料で使えること
6. ローカルで完結すること
7. ユーザーに確認すること
8. 再現性
9. 保守性

危険性が少しでもある場合は、実行せず、説明と提案に留める。
